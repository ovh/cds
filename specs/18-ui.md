# UI and websocket

This document specifies the Angular UI client of CDS and the
engine-side websocket server that pushes live events to the UI and
the CLI.

The CLI (`cdsctl`) is documented in [`15-cli.md`](./15-cli.md). The
Go SDK is in [`16-sdk.md`](./16-sdk.md); the gRPC plugin protocol in
[`17-plugins.md`](./17-plugins.md). The authentication backbone
(drivers, sessions, JWT) lives in [`08-auth.md`](./08-auth.md).

Source code anchors. The Angular UI lives under `ui/src/app/` (views,
NgRx store, services, models, shared components, route guards). The
engine-side websocket server is in `engine/api/v2_websocket.go`
(v2 user channel), `engine/api/websocket.go` (v1 channel), and
`engine/api/v2_hatchery_ws.go` (hatchery channel). The broker behind
them is in `engine/websocket/`.

## 1. Scope

**In scope** â€” Angular app top-level structure
(`ui/src/app/{views,store,service,model,shared,guard}/`); the 12
view modules; the 16 NgRx substates; the ~37 domain services; HTTP
interceptors; route guards; v1 websocket client (`event.service.ts`);
v2 websocket client (`event-v2.service.ts`) and filter union; the
engine-side `websocketV2Server`; the broker fan-out; the three
websocket endpoints `/ws`, `/v2/ws`, `/v2/hatchery/ws`.

**Out of scope** â€” cdsctl (see [`15-cli.md`](./15-cli.md)); Go SDK
(see [`16-sdk.md`](./16-sdk.md)); gRPC plugins (see
[`17-plugins.md`](./17-plugins.md)); auth drivers
(see [`08-auth.md`](./08-auth.md)); RBAC enforcement (see
[`09-rbac.md`](./09-rbac.md)); CDN log streaming websocket
(`/item/stream`, see
[`12-cdn-and-artifacts.md`](./12-cdn-and-artifacts.md)).

## 3. UI architecture

The application is at `ui/src/app/`:

| Directory | Purpose |
| --- | --- |
| `views/` | Routed pages and their child components |
| `store/` | NgRx state (~16 substates) |
| `service/` | HTTP / websocket / domain services (~37 services) |
| `model/` | TypeScript representations of CDS structs (~52 files) |
| `shared/` | Reusable components, directives, pipes |
| `guard/` | Route guards (auth, MFA, project loaded) |

Three top-level utilities sit beside these directories:

- `event-v2.service.ts` â€” v2 websocket client.
- `event.service.ts` â€” v1 websocket client.
- `app.service.ts` â€” dispatches incoming events to the right NgRx
  action.

Four HTTP interceptors round out the stack (under
`ui/src/app/service/authentication/`):

- `error.interceptor.ts` â€” surface API errors as toasts.
- `logout.interceptor.ts` â€” auto-logout on 401.
- `proxy.interceptor.ts` â€” rewrite URLs through the configured
  proxy.
- `xsrf.interceptor.ts` â€” XSRF cookie propagation.

## 8. Route guards

Route guards live in `ui/src/app/guard/`:

- `authentication.guard.ts` â€” enforce a valid session on protected
  routes.
- `no-authentication.guard.ts` â€” redirect already-signed-in users
  away from the login page.

Additional guards exist for specific UI invariants (e.g. ensuring a
project is loaded before opening a sub-route).

## 9. Websocket consumption

The v2 client lives at `ui/src/app/event-v2.service.ts`. It opens
`wss://host/cdsapi/v2/ws`, sends a list of `WebsocketV2Filter`
records, and receives `WebsocketV2Event`. The connection is wrapped
in an rxjs `webSocket()` subject with `openObserver.next` to re-send
the latest filter list on every reconnection.

### 9.1 Filter types

`WebsocketV2Filter` (`ui/src/app/model/websocket-v2.ts`) is a tagged
union:

| Filter type | Payload |
| --- | --- |
| `GLOBAL` | none â€” receive every event the caller has RBAC to see |
| `PROJECT` | `project_key` |
| `PROJECT_RUNS` | `project_runs_params` â€” narrow to specific run identifiers |
| `PROJECT_PURGE_REPORT` | `purge_report_id` â€” track a single purge |
| `QUEUE` | none â€” broadcast queue updates |

### 9.2 Auto subscription

`subscribeAutoFromPath(path)` reads the Angular router URL and
converts it to the appropriate filter. Navigating to `/project/KEY`
switches to `PROJECT { project_key: "KEY" }`; navigating into a run
page narrows further to `PROJECT_RUNS`.

### 9.3 Event dispatch

Once an event arrives:

1. The service dispatches `AddEventV2(event)` to NgRx.
2. `EventV2State.last` updates.
3. `AppService.manageEvent()` runs the side effects: refresh the
   project state, reload the run, append a log line, etc.

The reconnect policy uses
`retryWhen(errors => errors.pipe(delay(2000)))` so a network blip
recovers transparently.

### 9.4 V1 events

The v1 client lives at `ui/src/app/event.service.ts` and connects to
`/ws`. It uses the same SDK contract as
[`16-sdk.md`](./16-sdk.md#9-events-websocket-client):
`[]WebsocketFilter` payload with project / workflow / run
identifiers.

## 10. Engine-side websocket server

The server side at `engine/api/v2_websocket.go` mirrors the client
filter union. Each `websocketV2ClientData` carries an `AuthConsumer`
and a `webSocketV2Filters` set. The structures are:

| Struct | Purpose |
| --- | --- |
| `websocketV2Server` | One per API instance; tracks every connected client |
| `websocketV2ClientData` | Per-connection state (consumer + filter set) |
| `webSocketV2Filters` | Helper for filter matching (`HasOneKey`, `GetFirstByType`) |

Three side websockets coexist:

| Endpoint | Concern | File |
| --- | --- | --- |
| `/ws` | v1 events | `engine/api/websocket.go` |
| `/v2/ws` | v2 user events | `engine/api/v2_websocket.go` |
| `/v2/hatchery/ws` | hatchery channel | `engine/api/v2_hatchery_ws.go` |

A separate websocket from CDN (`engine/cdn/websocket.go`,
`/item/stream`) is used for live log streaming â€” see
[`12-cdn-and-artifacts.md`](./12-cdn-and-artifacts.md).

## 11. Broker

`engine/websocket/` holds the in-memory broker that does the
fan-out:

| File | Purpose |
| --- | --- |
| `broker.go` | `Broker` struct, `Init()`, `subscribe` (via `cache.PubSub`), `onMessage` callback |
| `server.go` | `Server`, `Client`, `UUID` |
| `client.go` | Client implementation: per-connection write loop |

When the API emits an event:

1. The event lands on the Redis pub/sub channel.
2. Every API replica receives it through `cache.PubSub`.
3. The broker walks every connected client.
4. For each client, it evaluates `webSocketV2Filters.HasOneKey(event)`.
5. If the filter matches, it runs the RBAC check on the client's
   `AuthConsumer` (see [`09-rbac.md`](./09-rbac.md)).
6. If both gates pass, it forwards the event to the client's
   websocket.

This per-event RBAC check is what allows a user to subscribe with the
`GLOBAL` filter and still only see events on resources they have
read rights on.
